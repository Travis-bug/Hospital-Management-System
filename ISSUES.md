# DEVELOPMENT HISTORY: Bug Fixes, Logic refactors, Reasoning, and Commit Mapping

| Error / Change & Classification                        | Commit Type | The "Why" (Reasoning)                                                                                                                                                                                         | The Fix                                                                                                                          |
| ------------------------------------------------------ | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| Identity GUID Parsed as Staff Int (Swagger 500)        | fix:        | ASP.NET Identity stores `NameIdentifier` as a string user ID, but API controllers were parsing it as the clinic domain staff integer, causing `FormatException` and 500 responses in authenticated endpoints like doctor schedule. | Added a claims transformation that resolves the authenticated Identity user into a domain `DomainUserId` and `PublicId`, then updated controllers to use those claims instead of parsing the raw Identity claim. |
| Create Endpoints Exposed Server-Generated IDs          | fix:        | Swagger create payloads were using EF entities or mismatched DTOs, so callers were being asked to provide fields like `PublicId` that must be generated by the server.                                     | Switched creation flows to DTO-driven contracts (`BookAppointmentDto`, revised `CreateVisitDto`) and kept `PublicId` generation inside models/services only. |
| Patient Enrollment Required Doctor Too Early           | fix:        | New patients were forced to provide a doctor assignment during enrollment even though the workflow allows doctor assignment later, which blocked valid create-patient requests.                              | Removed doctor assignment from enrollment DTO validation and added a dedicated patient doctor assignment endpoint/service flow for post-enrollment linking. |
| Appointment Schedule Failed on Invalid Doctor Input    | fix:        | Looking up doctor schedule with a bad `doctorPublicId` or an auth parsing failure surfaced as an internal server error instead of a proper client-facing API response.                                      | Added centralized exception-to-ProblemDetails mapping so missing records return 404, authorization failures return 403, and malformed requests return 400 instead of generic 500s. |
| Public ID Length Drift vs Database Column Length       | fix:        | Several entities generated `PublicId` values longer than the column length allowed, creating hidden insert failures in scheduling and staff assignment flows.                                               | Aligned generated ID lengths for appointments, visits, shifts, and manager records to fit their existing database column constraints. |
| Seeded Doctor Login Not Linked to Domain Doctor        | fix:        | The seeded ASP.NET Identity doctor account had no guaranteed matching `Doctor` row in the clinic database, so authenticated doctor workflows could still fail after login.                                  | Updated startup seeding to create and link a doctor profile to the seeded Identity account when the clinic-side record is missing. |
| Invalid InverseProperty Mapping (EF Core Error)        | fix:        | EF Core failed to create DbContext because navigation properties between Visit and Doctor were mismatched. The InverseProperty attribute referenced a property that did not exist or did not match correctly. | Added missing navigation property `ICollection<Visit> Visits` in Doctor and aligned `[InverseProperty("Doctor")]` on both sides. |
| DbContext Creation Failure                             | fix:        | The application could not instantiate ClinicContext due to invalid entity relationships and annotations.                                                                                                      | Corrected entity relationships and ensured all navigation properties referenced valid counterparts.                              |
| Misuse of Collection vs Single Entity (Visit Fetch)    | fix:        | Used a method returning a collection (`GetVisitsByIdAsync`) where a single Visit was expected, leading to incorrect null checks and logic flaws.                                                              | Replaced with `GetVisitByIdAsync` to properly retrieve a single entity.                                                          |
| Fake Null Validation on Collections                    | fix:        | Checking `if (collection == null)` always passed even when empty, causing invalid data to be treated as valid.                                                                                                | Updated logic to validate single entity or check `.Any()` for collections.                                                       |
| Inconsistent Naming of Service Methods                 | refactor:   | Method names like `GetVisitsById` implied single entity but returned collections, causing confusion and misuse.                                                                                               | Renamed methods to clearly distinguish between singular (`GetVisitByIdAsync`) and plural (`GetVisitsAsync`).                     |
| Security Risk: Trusting Frontend DoctorId              | fix:        | Backend accepted doctorId from request, allowing users to access other doctors' data.                                                                                                                         | Enforced doctor identity from authenticated user context instead of request parameters.                                          |
| Missing Access Control Enforcement                     | fix:        | Data filtering methods were used as if they enforced permissions, but they only filtered data.                                                                                                                | Moved authorization responsibility to controller/service layer using authenticated identity.                                     |
| Incorrect Billing Logic (Unpaid Detection)             | fix:        | Unpaid bills were determined using `Amount > 0` instead of payment status, leading to incorrect results.                                                                                                      | Updated logic to use `IsPaid == false` for unpaid bills.                                                                         |
| Incorrect Outstanding Balance Calculation              | fix:        | Outstanding balance included all positive amounts, even paid ones, resulting in inflated totals.                                                                                                              | Restricted calculation to unpaid bills only using `!IsPaid`.                                                                     |
| Misinterpretation of Negative Amounts                  | fix:        | Negative amounts were incorrectly considered unpaid instead of representing refunds or credits.                                                                                                               | Clarified domain logic: positive = charge, negative = adjustment/refund.                                                         |
| Overexposure of Internal IDs                           | refactor:   | Internal database IDs (e.g., PatientId) were used externally, posing security risks.                                                                                                                          | Introduced and used PublicId for external access and routing.                                                                    |
| Missing PublicId Generation Consistency                | fix:        | Some entities lacked consistent PublicId generation, breaking uniform access patterns.                                                                                                                        | Standardized PublicId generation using SecureIdGenerator across entities.                                                        |
| Dependency Loop Warning in Entity Relationships        | fix:        | EF Core detected circular navigation dependencies (e.g., Patient self-reference and multiple bidirectional links).                                                                                            | Corrected navigation mappings and ensured proper use of `[InverseProperty]` to break ambiguity.                                  |
| Incorrect Relationship Modeling (Test & Result)        | refactor:   | Diagnostic tests and results were initially combined, preventing correct lifecycle representation.                                                                                                            | Separated into `DiagnosticTest` and `TestResult` entities with proper foreign key relationship.                                  |
| Missing Validation for Foreign Key Existence           | fix:        | Services attempted to create dependent records (e.g., TestResult, Fee) without confirming parent existence.                                                                                                   | Added validation checks to ensure parent entities exist before creation.                                                         |
| Search Without Audit Logging (Compliance Gap)          | feat:       | Patient data access via search was not logged, violating compliance expectations.                                                                                                                             | Integrated audit logging for search operations.                                                                                  |
| Potential Sensitive Data Logging Risk                  | refactor:   | Logging raw search keywords could expose sensitive patient data.                                                                                                                                              | Adjusted logging strategy to minimize sensitive data exposure.                                                                   |
| Role-Based Access Logic Duplication                    | refactor:   | Repeated role-check logic across multiple methods increased maintenance complexity.                                                                                                                           | Identified for future refactor into shared helper methods.                                                                       |
| Incorrect Use of Amount in Payment Update              | fix:        | Payment update logic relied on amount conditions rather than explicit payment state.                                                                                                                          | Standardized payment updates using `IsPaid` flag.                                                                                |
| Improper Migration Execution Path                      | fix:        | Migration commands were executed from incorrect directory, causing failures or long execution times.                                                                                                          | Ensured commands run from correct project/solution directory.                                                                    |
| Entity Not Recognized by EF Core                       | fix:        | New entity was not registered or correctly configured, preventing migrations.                                                                                                                                 | Added entity to DbContext and ensured proper configuration.                                                                      |
| Overcomplicated Service Dependencies (Design Risk      | refactor:   | Service interactions and entity relationships created perceived dependency loops.                                                                                                                             | Clarified architecture: services depend only on DbContext, not each other.                                                       |
| Confusion Between Access Control and Filtering         | refactor:   | Filtering logic was mistaken for authorization logic, leading to insecure design assumptions.                                                                                                                 | Clearly separated access control (auth context) from filtering (query logic).                                                    |
| Lack of Clear API Design for Doctor vs Patient Context | refactor:   | Single methods attempted to handle both doctor-wide and patient-specific queries.                                                                                                                             | Redesigned approach to separate                                                                                                  |
| Complex Overlap Validation Logic | refactor: | Separating Date and Time into two columns made double-booking validation queries overly complex and inefficient for the database. | Merged Date and Time into a single AppointmentDate (DateTime) property, simplifying LINQ overlap checks. |
| Appointment Date Overwritten on Booking | fix: | BookAppointmentAsync was overwriting the user-provided appointment date with the current time (DateTime.Now), breaking the scheduling functionality. | Removed the overwrite logic so the requested date is preserved, and utilized a separate timestamp purely for the Audit Log. |
| Null Reference Risk in Appointment Retrieval | fix: | GetAppointmentByPublicIdAsync was typed to return a non-nullable Appointment but could return null if not found, potentially crashing the application. | Changed the return type to Appointment? to enforce explicit null handling at the call site. |
| Incorrect Exception Message in Cancelation | fix: | CancelAppointmentAsync threw an exception stating "Fee not found" due to a copy-paste error from the Billing service. | Corrected the exception message to accurately state "Appointment not found". |
| Clinical Data Truncation Risk | fix: | MySQL text columns (Symptoms, Diagnosis, Treatment, VisitNotes, Notes) had restrictive [StringLength(30)] annotations, preventing doctors from writing full clinical notes. | Removed the [StringLength(30)] annotations from free-text clinical fields to allow complete data entry. |
| Dead Code in AuditService | refactor: | The AuditService.LogAsync method contained an empty if block with a comment about safety checks that did nothing. | Removed the empty if block to clean up the code and reduce developer confusion. |
| Internal ID Leakage in Audit Logs (IDOR Risk) | fix: | VisitService and BillingService were logging the internal database PK (currentUserId) into plain-text audit log details, creating an enumeration risk if logs were exported. | Swapped {currentUserId} for the secure {actorPublicId} in all audit log Details strings. |
| Duplicate Role-Checking Logic (TODO) | chore: | Role validation if/else blocks are identical and duplicated across 12+ methods, reducing code maintainability. | Logged in backlog for future refactor to extract into a shared AuthorizeRole(string role, params string[] allowedRoles) helper method. |
| Method Naming & Comment Typos (TODO) | chore: | PatientperimeterValidation has a typo and confusing naming; comments contained unprofessional typos ("Relationshitp", "mnight"). | Logged in backlog to rename method to ValidateEnrollmentAsync and correct documentation spelling. |
| Swagger Appointment Read Failed With Object Cycle | fix: | `GET /api/Appointment/{publicId}` returned a raw partial payload and then failed because the API was serializing EF entities with bidirectional navigation properties (`Appointment -> Patient -> Appointments -> Patient ...`), which caused a JSON cycle during Swagger rendering. | Added MVC JSON serializer cycle handling in `Program.cs` so entity graphs no longer recurse infinitely during API serialization. |
| Patient Read Returned 500 During Audit Save | fix: | `GET /api/Patient/{publicId}` could successfully find the patient but still return 500 because the follow-up audit insert overflowed the current `AuditLog.Details varchar(50)` column, causing `SaveChangesAsync` to fail on read operations. | Hardened `AuditService` to sanitize and truncate audit field values centrally before insert so read auditing no longer crashes the request pipeline. |
| Appointment API Still Returns Full EF Entity Graph (TODO) | chore: | Appointment read endpoints still return the full entity model, including nested patient data and navigation collections, which is noisy in Swagger and leaks more structure than the client needs. | Reminder: switch appointment responses to dedicated read DTOs/list DTOs instead of returning `Appointment` entities directly. |
| Dockerfile Targeted Old Project Name and Runtime | fix: | The container build still referenced `Clinic-Management` and .NET 10 even though the shipped app is `Hospital-Management-System` on .NET 9, so container builds would fail immediately. | Updated the Dockerfile to restore, build, publish, and run `Hospital-Management-System.csproj` on the .NET 9 images. |
| Billing State Logic Used Amount Instead of Payment State | fix: | Billing methods still used `Amount > 0` to determine unpaid state, which made unpaid totals incorrect and blocked state transitions for zero or refund-style records. | Switched unpaid-bill, outstanding-balance, and mark-paid logic to the explicit `IsPaid` flag instead of deriving payment state from amount. |
| Audit Logs Still Leaked Internal IDs and Search Keywords | fix: | Billing and test-result logs still recorded internal PKs and raw search keywords, leaving the earlier public-ID/privacy cleanup incomplete. | Replaced those audit details with public-ID-safe messages and count-based search logs that preserve traceability without storing sensitive search text. |
| AuditLog Details Column Was Too Small for Real Events | fix: | `AuditLog.Details` at `varchar(50)` was too short for meaningful operational/compliance events and had already caused read endpoints to fail on overflow. | Increased the model and service cap to 255 characters and added a migration to expand the database column. |
| Appointment Booking Still Returned Raw EF Entity | fix: | Appointment reads had been converted to DTOs, but the booking endpoint still returned the raw EF entity shape, exposing implementation details unnecessarily. | Changed appointment booking to return the same public-safe appointment detail DTO used by the read endpoint. |
| Staff Provisioning Lacked Real Activation Flow | feat: | Provisioned staff accounts existed, but there was no actual activation path to confirm email, replace the temporary password on the same `IdentityUser`, and complete onboarding cleanly. | Added SMTP-backed activation emails, a token-based `complete-onboarding` endpoint, and a dedicated SPA activation screen that confirms email and resets the password on the existing account. |
| Two-Factor Login Was Not Actually Emailing Codes | fix: | The 2FA bridge generated email tokens but only logged them in development, so the sign-in flow was incomplete outside the console. | Wired the login flow to the SMTP sender so 2FA codes are delivered by email instead of only being logged. |
| Startup Identity Patcher Caused Seed Drift | refactor: | `StaffIdentityPatcher` kept mutating Identity state on startup, which made seeded-account behavior harder to reason about once explicit staff provisioning existed. | Retired the runtime patcher, removed it from startup, and moved the legacy source out of the active project so staff onboarding is the authoritative account-creation path. |
| Auth Cookie Lifetime Was Implicit | security: | The app relied on default cookie timing, which made session duration ambiguous and weaker than a deliberate clinical-session policy. | Set the ASP.NET Identity application cookie to a 1-hour lifetime with sliding expiration. |
| Local Development Still Mixed HTTP and HTTPS | security: | The frontend dev server, activation links, and backend launch profiles were split across HTTP and HTTPS, which left the local auth path inconsistent and susceptible to plain-HTTP transport. | Standardized the dev transport to HTTPS by switching Vite to a local TLS certificate, proxying to the backend HTTPS endpoint, moving the backend launch profiles to HTTPS, and tightening CORS to HTTPS frontend origins only. |
| Cookie Secure Transport Was Environment-Conditional | security: | The auth cookie only used `SecurePolicy.Always` outside development, which meant the dev auth session could still ride over plain HTTP. | Forced `CookieSecurePolicy.Always` and enabled HTTPS redirection for all environments so cookies are sent only over HTTPS. |
| Activation Emails Could Target the Wrong Host | fix: | Activation links were built from the current request origin, which broke onboarding whenever the API host and the SPA host differed. | Added configurable frontend base-URL resolution for activation links, so email onboarding targets the intended SPA host instead of assuming the API origin. |
